LoRa and LoRaWAN Overview

What is each component in LoRaWAN?

  • End Device: A sensor or actuator that connects wirelessly to a LoRaWAN network using LoRa modulation.
  • Gateway: Receives LoRa RF messages from end devices and forwards them to the network server, connected via an IP backbone.
  • Network Server (LNS): Manages network parameters, establishes secure connections, and ensures the authenticity and integrity of messages and devices.
  • Application Server: Handles, manages, and interprets sensor application data, generating application-layer downlink payloads.
  • Join Server: Manages the over-the-air activation process, handling join-request and join-accept messages, and session key derivations.

What is the Join Procedure in LoRaWAN?

The Join Procedure in LoRaWAN involves a two-step process for network access: Join Request and Join Accept. The end device initiates the process by sending a Join Request to the network, which includes device identifiers. The Join Server authenticates this request using pre-shared keys and responds with a Join Accept message if successful. This process establishes mutual authentication and enables the derivation of session keys (NwkSKey for network and AppSKey for application), securing subsequent communications between the end device and the network.

How does LoRaWAN protect each data packet?

LoRaWAN protects each data packet through encryption and integrity checks. Each packet is encrypted using the AES algorithm with two types of keys: the Network Session Key (NwkSKey) for network layer security and the Application Session Key (AppSKey) for application payload encryption. This ensures that the content is confidential and cannot be tampered with during transmission. Additionally, a unique frame counter is used for each packet to protect against replay attacks, ensuring that each packet is fresh and hasn’t been intercepted and resent by malicious actors.

End Device

A sensor or an actuator is wirelessly connected to a LoRaWAN network through radio gateways using LoRa modulation.

  • Examples: street lighting, wireless locks, water valve shut off, leak prevention.
  • Several unique identifiers were assigned during the manufacture.
    • Identifiers are used to securely activate and administer the device to ensure the safe transport of packets over a private or public network and to deliver encrypted data to the Cloud.

![image-20240219181813957](/Users/alexander/Library/Application Support/typora-user-images/image-20240219181813957.png)

Gateway

Receives LoRa-modulated RF messages from any end device in hearing distance and forwards these data messages to the LoRaWAN network server (LNS), connected through an IP backbone.

  • The same sensor can be served by multiple gateways in the area.
  • Each uplink packet sent by the end device will be received by all gateways within reach.

![image-20240219181831807](/Users/alexander/Library/Application Support/typora-user-images/image-20240219181831807.png)

IP traffic from a gateway to the network server:

  • Wi-Fi
  • hardwired Ethernet
  • Cellular

LoRaWAN gateways perform at the physical layer like LoRa radio message forwarders.

  • Uplinks: Only check the data integrity of each incoming LoRa RF message. Check the CRC to see if the integrity is intact or not.
    • If incorrect, the message will be dropped.
    • If correct, the message will be forwarded to the LNS with metadata (receive the RSSI level of the message and optional timestamp).
  • Downlinks: Execute transmission requests from LNS without interpretation of the payload.
    • LNS performs the data de-duplication and copy deletion.
    • LNS selects the transmission gateway (the one that is the closest to the target end device) based on the RSSI level of the identical messages.

![image-20240219184305654](/Users/alexander/Library/Application Support/typora-user-images/image-20240219184305654.png)

There are three types of channel gateways in North America.

  • 8-channel: least expensive.
  • 8- and 16-channel: for both indoor and outdoor use.
  • 64-channel: only available in a carrier-grade variant (intended for deployment in places such as cell towers, or the rooftops of very tall buildings).
Network Servers

The LNS establishes secure 128-bit AES connections

  • Transport of the end-to-end data: end-device User Application in the Cloud
  • Control of traffic: end-device ​ LNS

Also ensures the authenticity of every sensor on the network and the integrity of every message. The network server cannot see or access the application data.

![image-20240220004322448](/Users/alexander/Library/Application Support/typora-user-images/image-20240220004322448.png)

Features:

  • Device address checking
  • Frame authentication and frame counter management
  • Acknowledgements of received messages
  • Adapting data rates using the ADR protocol
  • Responding to all MAC layer requests coming from the device
  • Forwarding uplink application payloads to the appropriate application servers
  • Queueing of downlink payloads coming from any Application Server to any device connected to the network
  • Forwarding Join-request and join-accept messages between the devices and the join server
Application Servers

Responsible for securely handling, managing, and interpreting sensor application data.

Also, generate all the application-layer downlink payloads to the connected end devices.

![image-20240220004626050](/Users/alexander/Library/Application Support/typora-user-images/image-20240220004626050.png)

Join Server

  • Manages the over-the-air activation process for end devices to be added to the network.

  • Contains the information required to process uplink join-request frames and generate the downlink join-accept frames.

  • Signals to the network server which application server should be connected to the end-device, and performs the network and application session encryption key derivations.

  • Communicates the Network Session Key of the device to the network server, and the Application Session Key to the corresponding application server.

![image-20240220005916368](/Users/alexander/Library/Application Support/typora-user-images/image-20240220005916368.png)

Information contained in the join server for end-device control:

  • DevEUI (end-device serial unique identifier)
  • AppKey (application encryption key)
  • NwkKey (network encryption key)
  • Application Server Identifier
  • End-device Service Profile

Device Commissioning

Devices must be commissioned and activated on the network at the start of operation (align each device and the network with respect to parameters like identifiers/encryption keys/server locations).

There are two activation types: Over-the-air Activation (OTAA) and Activation by Personalization (ABP).

OTAA (Preferred) ABP
- Device manufacturers autonomously generate essential provisioning parameters
- Secure keys (session-long and derived) can be renewed regularly
- Devices can store multiple “identities” to dynamically and securely switch networks and operations during their lifetime
- High-grade, tamper-proof security options are available		 - A simplified (less secure) commissioning process
- IDs and Keys are personalized at fabrication
- Devices become immediately functional upon powering up; the Join procedure is skipped
- Devices are tied to a specific network/service; the NetID is a portion of the device network address

Security

Two key elements to the security of a LoRaWAN network:

  • Join Procedure: Establish mutual authentication between an end device and the LoRaWAN network to which it is connected.
  • Message Authentication: LoRaWAN MAC and application messages are origin-authenticated, integrity-protected, and encrypted end-to-end (i.e. end-device ​ application server)

Key features:

  • No network traffic alteration
  • No non-legitimate devices connected
  • No eavesdropping
  • No network traffic capture/replay

Device Classes: A, B and C

  • Class A: supported by all devices
  • Class B: support both Class A and Class B modes
  • Class C: support all three modes of operation
Class A Devices

![image-20240220200910912](/Users/alexander/Library/Application Support/typora-user-images/image-20240220200910912.png)

End-device mostly stays in an idle state (sleep mode). It wakes up and initiates an uplink when there is a change in the monitored target, transmitting the data about the changed state back to the network (Tx).

Then waiting for the network response (typically 1 second).

  • Doesn’t receive a downlink during the receive window (Rx1)
    • Briefly goes back to sleep
    • Waking a moment later, and listening for a response (Rx2)
  • No response received during the second Rx window
    • Goes back to sleep until next data report

Class A device cannot be waken up by the application.

![image-20240220200924446](/Users/alexander/Library/Application Support/typora-user-images/image-20240220200924446.png)

A device will nor try to send another uplink message until either:

  • It has received a downlink message during Rx1
  • The second receive window following the last tranmission is complete
Class B Devices

An enhancement of Class A, offering regularly-scheduled, fixed-time opportunities for an end device to receive downlink from the network. Making it suitable for

  • monitoring sensors
  • actuators

All LoRa-based end-devices start in Class A mode, but programmed Class B device may be switched to Class B mode by the application layer.

Apart from opening a receive window in Class A mode, Class B end-devices provide regularly-scheduled receive windows.

1.1.1.1 Class B Beacons

Beaconing process is required for the Class B mode to communicate with the network.

A time-synchronized beacon will be broadcasted periodically by the network via the gateways, and the end-device must periodically receive one og thses network beacons to align internal timing reference with the network.

![image-20240220225630534](/Users/alexander/Library/Application Support/typora-user-images/image-20240220225630534.png)

Not every beacon needs to be processed by the device if it is already aligned.

![image-20240220225815198](/Users/alexander/Library/Application Support/typora-user-images/image-20240220225815198.png)

End devices can open receive windows (ping slots) periodically based on the beacon’s timing reference, and these ping slots can be used by the network infrastructure to initiate a downlink communication.

All LoRaWAN gateways have a built-in GPS timing source to synchronize with the exact beacon timing.

![image-20240220230346074](/Users/alexander/Library/Application Support/typora-user-images/image-20240220230346074.png)

Class C Devices

Class C are always on devices.

street lights, eletrical meters etc.

It implemented the same two receive windows as Class A device, but do not close Rx2 window until sending the next transmission back to the server.

  • Receiving a downlink is always accessible during the Rx2 window.

  • A short window is also available between the end of a transmission and the begining of Rx1.
