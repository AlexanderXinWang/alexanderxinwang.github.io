The Art, Science, and Engineering of Fuzzing

1 Introduction

Fuzzing is used to

  • test software correctness and reliability
  • discover vulnerabilities before attackers do
  • gauge the security of commodity software packages and provide some suitable forms of assurance to end-users

Drawbacks

  • some fuzzers do not go much beyond their source code and manual page
  • fragmentation in the terminology ysed by various fuzzers

2 Systemization, Taxonomy, and Test Programs

2.1 Fuzzing & Fuzzing Test

Fuzzing is the action of running a Program Under Test (PUT) with fuzz inputs.

Fuzz input is an input that the PUT may not be expecting.

  • Fuzzing

    the execution of the PUT using input(s) sampled from an input space (the fuzz input space) that protrudes the expected input space of the PUT.

    • Fuzz input space doesn't need to contain the expected input space.
    • Fuzzing runs for many iterations in practice. (repeated executions)
    • The sampling process is not necessarily randomized.

  • Fuzz Testing

    the use of fuzzing to test if a PUT violates a correctness policy.

  • Fuzzer

    a program that performs fuzz testing on a PUT.

  • Fuzz Campaign

    a specific execution of a fuzzer on a PUT with a specific correctness policy.

    • To find bugs that violate the specified correctness policy.

  • Bug Oracle

    a program/part of a fuzzer, that determines whether a given execution of the PUT violates a specific corrrectness policy.

  • Fuzz Configuration

    A fuzz configuration of a fuzz algorithm comprises the parameter values(s) that control(s) the fuzz algorithm.

    • The type of values in a fuzz configuration depends on the type of the fuzz algorithm.
    • Sophisticated fuzzers contain algorithms that accept a set of configurations and evolve the set over time—this includes adding and removing configurations.

2.4 Taxonomy of Fuzzers

Categorize fuzzers into three groups based on the granularity of semantics the fuzzer observes in each fuzz run.

2.4.1 Black-Box Fuzzer
  • Do not see the internals of the PUT
  • Observe only the input/output behavior of the PUT
2.4.2 White-Box Fuzzer
  • Generate test cases by analyzing the internals of the PUT and the information gathered when executing the PUT.
  • Explore the state space of the PUT systematically.
  • Used to describe fuzzers that employ taint analysis.
2.4.3 Grey-Box Fuzzer

  • Obtain some information internal to the PUT and/or its executions.

  • Perform lightweight static analysis on the PUT and/or gather dynamic information about its execution.

  • Rely on approximated, imperfect information in order to gain speed and be able to test more inputs.

3 Preprocess

Some fuzzers have a first step to prepare the main loop of Algorithm 1 by modifying the initial set of fuzz configurations before the first fuzz iteration.

  • instrument the PUT
  • weed out potentially redundant configurations
  • prepare a model for future input generation

3.1 Instrumentation

The method that collects the most valuable feedback.

  • Static Instrumentation
    • perform at compile time on either source code or intermediate code
    • less runtime overhead
  • Dynamic Instrumentation
    • perform at runtime
    • easily instrument dynamically-linked libraries
3.1.1 Execution Feedback

  • Typically used by grey-box fuzzers to evolve test cases.

  • Current Fuzzer

    • LibFuzzer and AFL

      Instrument every branch instruction in the PUT.

      Path collisions can occur.

    • CollAFL

      Introduce a new path-sensitive hash function to avoid path collisions.

    • Syzkaller

      Use node coverage as execution feedback.

    • honghfuzz

      Allow users to choose which execution feedback to use.

3.1.2 Thread Scheduling

Instrumentation can be used to trigger different non-deterministic program behaviors by explicitly controlling how threads are scheduled (to find race condition bugs).

3.1.3 in-Memory Fuzzing

Sometimes it is desirable to fuzz only a portion of the PUT without spawning a process for each fuzz iteration in order to minimize execution overhead.

  • complex (e.g., GUI) applications
    • take a snapshot
    • restore the memory snapshot before writing the new test case directly into memory and executing it
  • in-memory fuzzing
    • create a snapshot before loading any input types
    • side effects
      • found bugs may not be reproducible
    • depends on a good entry point function

  • Smaller seeds are likely to consume less memory and entail higher throughput.

  • Perform only once at the beginning of a fuzz campaign

4 Scheduling

Scheduling: selecting a fuzz configuration for the next fuzz iteration.

4.1 The Fuzz Configuration Scheduling (FCS) Problem

The goal of scheduling

  • analyze the currently available information about the configurations
  • pick one that is more likely to lead to the most favorable outcome

exploration versus exploitation conflict (FCS)

  • gathering more accurate information on each configuration to inform future decisions
  • fuzzing the configurations that are currently believed to lead to more favorable outcomes

4.2 Black-Box FCS Algorithm

Fuzz outcomes of a configuration are the only information that an FCS algorithm can use.

  • the number of crashes and bugs found with it
  • the amount of time spent on it so far

Householder and Foote: how such information can be leveraged in the CERT BFF black-box mutational fuzzer.

Woo: further improved on multiple fronts.

4.3 Grey-Box FCS Algorithm

Use a richer set of information about each configuration.

AFL: based on an evolutionary algorithm (EA).

  • maintain a population of configurations, each with some value of "fitness"
  • EA
    • select fit configuration
    • apply them to genetic transformations such as mutation and recombination to produce offspring new configurations

FCS in the context of an EA

  • First need to define:

    • what makes a configuration fit

    • how configurations are selected

    • how a selected configuration is used

  • On high-level approximation

    • Consider the one that contains the fastest and smallest input to be fit.

    • Maintain a queue of configurations

      selects the next fit configuration essentially as if the queue is circular

    • Once selected, allocate more runs to configurations that are fastest and have a higher branch coverage.

AFLFast

  • Modify configuration fitness setting and selection to prioritize exploration of new and rare paths.

  • Fuzze a selected configuration a variable number of times as determined by the power schedule.

    A power schedule is a technique used in fuzzing algorithms to determine the number of times a selected configuration is fuzzed during an iteration. AFLFast, for example, implements a power schedule that starts with a small "energy" value to encourage initial exploration among configurations and then exponentially increases this energy value up to a limit to ensure sufficient exploitation of the configurations. This approach helps balance the exploration of new and rare paths with exploiting configurations that have shown promise in finding bugs

  • Normalize the energy by the number of generated inputs that exercise the same path.

5 Input Generation

Traditionally

  • generation-based: produce test cases based on a given model that describes the inputs expected by the PUT.
    • grammar characterizing the input format
    • magic values identifying filetypes
  • mutation-based: produce test cases by mutating a given seed input.
    • generally considered model-less

5.1 Model-based (Generation-based) Fuzzers

5.1.1 Predefined Model
  • Specify the number and types of arguments a system call expects as inputs.
  • Others target a specific language or grammar, and the model of this language is built into the fuzzer itself.
5.1.2 Inferred Model

Without relying on a predefined or user-provided model.

  • In : infer the model as a preprocessing step.
  • In : update the model after each fuzz iteration.
5.1.3 Encoder Model

  • Compared to the decoder program, encoder programs can be thought of as an implicit model of the file format.

  • The mutation of the encoder program can produce new test cases.

5.2 Model-less (Mutation-based) Fuzzers

  • Classic random testing is not efficient in generating test cases that satisfy path conditions.
  • A seed is typically a well-structured input of a type supported by the PUT: a file, a network packet, or a sequence of UI events.
5.2.1 Bit-Flipping

Some fuzzers simply flip a fixed number of bits, while others determine the number of bits to flip at random.

mutation ratio, which determines the number of bit positions to flip for a single execution of INPUTGEN.

5.2.2 Arithmetic Mutation
  • Consider a selected byte sequence as an integer and perform simple arithmetic on that value
  • The computed value is then used to replace the selected byte sequence.
5.2.3 Block-based Mutation

Block: a sequence of bytes of a seed.

  • insert a randomly generated block into a random position of a seed
  • delete a randomly selected block from a seed
  • replace a randomly-selected block with a random value
  • randomly permute the order of a sequence of blocks
  • resize a seed by appending a random block
  • take a random block from a seed to insert/replace a random block of another seed
5.2.4 Dictionary-based Mutation

Use a set of predefined values with potentially significant semantic meaning for mutation.

5.3 White-box Fuzzers

It can also be categorized into either model-based or model-less fuzzers.

  • traditional symbolic execution does not require any model as in mutation-based fuzzers
  • some symbolic executors leverage input models such as grammar to guide
5.3.1 Dynamic Symbolic Execution
  • Build symbolic expressions instead of evaluating concrete values.
  • For every path, a symbolic interpreter builds up a path formula (or path predicate) for every branch instruction it encounters during an execution.
  • A variant of traditional symbolic execution, where both symbolic execution and concrete execution operate at the same time. concolic (concrete + symbolic)
    • Concrete reduces the complexity of symbolic constraints.
  • Execution is slow compared to greybox or blackbox approaches. (instrumenting and analyzing every instruction of the PUT)
    • let the user specify uninteresting parts of the code
    • alternate between concolic testing and greybox fuzzing
5.3.2 Guided Fuzzing

Leverage static or dynamic program analysis techniques to enhance the effectiveness of fuzzing.

  • a costly program analysis for obtaining useful information about the PUT
  • test case generation with the guidance from the previous analysis
5.3.3 PUT Mutation

Bypassing a checksum validation is one of the practical challenges in fuzzing.

  • checksum-aware fuzzing technique
    • Identify a checksum test instruction with a taint analysis
    • Patch the PUT to bypass the checksum validation
    • Generate the correct checksum for the input to generate a test case that crashes the unmodified PUT (when finding a program crash)
  • Stitched dynamic symbolic execution: generate test cases in the presence of checksums.
  • T-Fuzz
    • Non-Critical Checks: a set of branches that can be transformed without modifying the program logic.
    • Pick an NCC when the fuzz campaign stops discovering new paths
    • Restart a fuzz campaign with fuzzing on a modified PUT
    • After a crash is found, reconstructed on the original PUT with symbolic execution.

6 Input Evaluation

Evaluating the resulting execution of generated inputs.

6.1 Bug Oracles

  • Consider every program execution terminated by a fatal signal to be a violation.
  • Some memory vulnerabilities may not be detected.

Sanitizers: a variety of efficient program transformations to detect unsafe or unwanted program behaviors and abort the program.

6.1.1 Memory and Type Safety

Memory safety errors:

  • spatial: a pointer is dereferenced outside of the object it was intended to point to.
  • temporal: a pointer is accessed after it is no longer valid.
6.1.2 Undefined Behaviors

Languages such as C contain many behaviors that are left undefined by the language specification. The compiler is free to handle these constructs in a variety of ways.

  • Memory Sanitizer: instruments programs during compilation to detect undefined behaviors caused by uses of uninitialized memory in C and C++.
  • Undefined Behavior Sanitizer: modifies programs at compile-time to detect undefined behaviors.
  • Thread Sanitizer: a compile-time modification that detects data races.
6.1.3 Input validation

Testing for input validation vulnerabilities such as XSS (cross-site scripting) and SQL injection vulnerabilities is a challenging problem.

  • KameleonFuzz: detects successful XSS attacks by parsing inputs with a real web browser, extracting the Document Object Model tree, and comparing it against manually specified patterns that indicate a successful attack.
  • $$4SQLi: uses a database proxy that intercepts communication between the target web application and the database to detect whether an input triggered harmful behavior.
6.1.4 Semantic Difference

Semantic bugs are often discovered using a technique called differential testing, which compares the behavior of similar (but not identical) programs.

6.2 Execution Optimizations

Modern fuzzers provide functionalities to skip the repetitive loading processes.

  • AFL: fork server

6.3 Triage

The process of analyzing and reporting test cases that cause policy violations.

6.3.1 Deduplication

The process of pruning any test case from the output set that triggers the same bug as another test case.

  • Avoid wasting disk space and other resources by storing duplicate results on the hard drive.
  • Easy for users to understand roughly how many different bugs are present and to be able to analyze an example of each bug

Implementation

  • stack backtrace hashing: an automated tool records a stack backtrace at the time of the crash and assigns a stack hash based on the contents of that backtrace.
  • coverage-based deduplication: record the edge coverage of each execution of the PUT and measure coarse hit counts for each edge.
  • semantics-aware deduplication: based on the semantics recovered from a reverse data-flow analysis on each crash.
6.3.2 Prioritization and Exploitability

prioritization (the fuzzer taming problem)

  • The process of ranking or grouping violating test cases according to their severity and uniqueness.
  • Determining the exploitability of a crash.
6.3.3 Test Case Minimization

The process of identifying the portion of a violating test case that is necessary to trigger the violation, and optionally producing a test case that is smaller and simpler than the original but still causes a violation

7 Configuration Updating

distinguishing the behavior of black-box fuzzers from grey- and white-box fuzzers.

7.1 Evolutionary Seed Pool Update

An Evolutionary Algorithm (EA) is a heuristic-based approach that involves bio-inspired evolution mechanisms such as mutation, recombination, and selection.

seed pools of different sizes can have the same coverage

A common strategy in EA fuzzers is to refine the fitness function so that it can detect more subtle and granular indicators of improvements.

Another common strategy is to measure the fraction of conditions that are met when complex branch conditions are evaluated.

7.2 Maintaining a Miniset

Miniset is used to mitigate the risk of creating too many configurations.

a minimal set of test cases that maximizes a coverage metric.
